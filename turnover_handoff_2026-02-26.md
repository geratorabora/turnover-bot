# Turnover Bot / DataLens — промежуточный статус (handoff)

Дата: 2026-02-26  
Проект: автоматизация дашбордов (бот загрузки Excel → PostgreSQL → витрина для DataLens)

---

## 1) Текущий статус

### 1.1 Бот
- Бот принимает Excel-файл, парсит строки и грузит в PostgreSQL (Railway).
- Исправлены проблемы:
  - `NaN` больше не ломает вставку в БД (числа приводим к `NULL`).
  - Договорились о **строгом контракте имён колонок в БД**, как во второй строке Excel-файла с маппингом.
- Подход к изменениям кода:
  - **Код делим на блоки** `1A, 1B, 1C, 2A, 2B, 2C, 3A, 3B, 3C...`
  - **Меняем только целиком блоками**, блоки не должны быть слишком большими.
  - **Код комментировать построчно** (пользователь новичок).
  - Общение — **пошагово**, с объяснением каждого шага.

### 1.2 База данных (Railway Postgres)
- Основная таблица: `public.raw_turnover_stock`
- После перезагрузки истории успешно загружены недели, проверено агрегатами.
- Были «старые» периоды (2011/2018) — **исправлены обновлением `period`**, теперь всё ОК.

### 1.3 Витрина/VIEW для сводной (pg → segment)
Создана вьюха, которая отдаёт 3 метрики по каждой неделе (`period`) на уровне `(pg, segment)`:

`public.v_pg_segment_week_metrics`
- `stock_cost_sum` = `sum(curr_stock_cost)`
- `ob_rub` = `round(sum(sales_cost) / nullif(sum(av_stock_cost), 0), 2)`
- `stock_cost_turns_lt_2` = `sum(curr_stock_cost) filter (where turns_rub < 2)`
- Отсекаем строки: `having sum(curr_stock_cost) >= 100000`

Проверка выборкой показала, что данные «вразумительные», метрики считаются.

## 2) Проблема с DataLens

### 2.1 Прямое подключение DataLens → Railway Postgres не работает
Ошибка:
- `SSL error: certificate verify failed`
- В DataLens есть только переключатель `TLS вкл/выкл + CA file`, без выбора `sslmode`.
- `PGHOST` = `postgres.railway.internal` (внутренний, не подходит внешним клиентам).
- В Railway Settings виден только TCP proxy:
  - `yamanote.proxy.rlwy.net:31212 -> :5432`
- Прикрепление CA (ISRG Root X1) **не помогло**, verify всё равно падает.
- `openssl` на ПК пользователя не установлен, поэтому быстро вытащить цепочку сертификатов не получилось.

---

## 3) Принятое решение

### 3.1 Переходим на Supabase (Free plan)
Причина: нужен нормальный внешний Postgres endpoint с корректным SSL, чтобы DataLens подключался напрямую без плясок с proxy/сертификатами.

---

## 4) Ближайшие задачи (чек-лист)

### 4.1 Supabase: старт
1) Создать проект Supabase (Free).
2) Получить параметры подключения:
   - host, port (обычно 5432), db name, user, password, SSL.

### 4.2 Перенос схемы
Перенести (минимум):
- `public.raw_turnover_stock` (DDL + индексы, если есть уникальные)
- `public.v_pg_segment_week_metrics` (VIEW)

Лучший путь: сгенерировать DDL из DBeaver для таблицы/вьюхи и выполнить в Supabase.

### 4.3 Перенос данных
Варианты (выбрать самый простой):
- А) Перезалить историю через бота, но уже в Supabase (меняем `DATABASE_URL`, `truncate`, загружаем 6–8 недель Excel).
- B) Экспорт/импорт CSV из DBeaver.
- C) pg_dump/restore (если захотим позже).

### 4.4 Перенастроить бота
- Обновить переменные подключения к БД (Supabase connection string).
- Деплой на Railway.
- Проверка командой `/db` (доступ/таблица/права).

### 4.5 Подключить DataLens напрямую к Supabase Postgres
- Создать подключение PostgreSQL в DataLens.
- Проверить доступ к `v_pg_segment_week_metrics`.
- Собрать сводную:
  - Строки: `pg` → `segment`
  - Колонки: `period` (позже можно обсудить номер недели)
  - Метрики: `stock_cost_sum`, `ob_rub`, `stock_cost_turns_lt_2`
  - Сортировка строк по убыванию `stock_cost_sum` (обычно по последней неделе).

## 5) Примечания по стилю работы (обязательно)

- Пользователь учится и новичок:
  - объяснять каждый шаг,
  - давать короткие команды «сделай → покажи результат → идём дальше».
- Код:
  - построчные комментарии,
  - не вставлять «кусочками» — только блоками `1A/1B/...`,
  - блоки небольшие, чтобы избегать ошибок с отступами/запятыми.
- Пользователь обещал: текущий `main.py` перенесёт в новое окно проекта.
