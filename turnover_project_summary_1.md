# Turnover & Stock Dashboard Pipeline — Project Summary (для переноса в новый чат)

Дата: 2026‑02‑25  
Проект: **turnover-bot** (Telegram бот для загрузки weekly Excel‑отчёта в PostgreSQL на Railway → далее DataLens дашборды по остаткам/оборачиваемости)

---

## 1) Итоговая цель

Автоматизировать путь **“еженедельный отчёт из почты (Excel) → база данных → дашборды в DataLens”**:

- В Telegram боту пересылается/отправляется Excel‑файл отчёта (weekly).
- Бот читает файл, нормализует и загружает данные в **PostgreSQL (Railway)**.
- В DataLens строятся дашборды:
  - текущее состояние (остатки/резервы/неликвид и т.п.)
  - динамика по периодам (снимки по `Period`/`report_ts`)
  - оборачиваемость и другие метрики (позже)

---

## 2) Архитектура (как сейчас)

### Компоненты
1) **Telegram Bot** (aiogram)  
   - получает файл `.xlsx`
   - читает pandas/openpyxl
   - грузит в PostgreSQL

2) **Railway**
   - сервис **worker** (Python) для бота
   - сервис **Postgres** (общая БД проекта)

3) **DBeaver**
   - клиент для подключения к Railway Postgres
   - проверка таблиц/данных/индексов, отладка SQL

### Поток данных
Excel → Telegram → Railway worker (бот) → Railway Postgres → DBeaver/DataLens

---

## 3) Репозиторий и деплой

- GitHub repo: `geratorabora/turnover-bot`
- Railway подключён через GitHub App (репозиторий выбран в “Repository access”)
- Деплой идёт автоматически по `git push` в ветку `main`

**Важно про polling (getUpdates):**
- В режиме polling одновременно может работать только **одна** копия бота.
- Если одновременно запущен бот локально и на Railway → ошибка:
  `TelegramConflictError: terminated by other getUpdates request`
- Правило: **либо Railway worker, либо локальный запуск**, не одновременно.

## 6) Договорённости по стилю работы

### 6.1 Код комментируем построчно
Требование пользователя: **каждая строка кода должна быть прокомментирована**, потому что пользователь учится.

### 6.2 Меняем код только “блоками”
Чтобы не уставать от ручных правок и путаницы:
- `main.py` будет размечен блоками: **1A, 1B, 1C, 2A...**
- Каждому блоку соответствует “назначение” в оглавлении.
- Любое изменение делаем так: **“замени блок 3B целиком на новый”**.

---

## 7) Эталонный размеченный main.py (план)

Принято решение перейти на версию, где:
- “важные” поля (для аналитики) пишутся в отдельные колонки таблицы
- **все поля отчёта** пишутся в `payload jsonb` (это позволяет не ломать загрузку при изменении колонок в отчёте)
- таблица/колонки создаются/расширяются через `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`
- загрузка делается через `INSERT ... ON CONFLICT ... DO UPDATE`

**Файл‑эталон:** `docs/main_marked.md` (в репозитории)  
Он содержит:
- оглавление блоков (1A..6B)
- полный `main.py` внутри ```python ... ```

## 9) Типичные проблемы и решения

### 9.1 TelegramConflictError
Причина: две копии polling (локально + Railway).
Решение: остановить локального бота (`Ctrl + C`) или выключить worker на Railway.

### 9.2 “Бот молчит после деплоя”
Причины:
- деплой не подтянул новый коммит
- worker упал при старте
Решение:
- Railway → worker → Deploy logs: проверить `Start polling`, ошибки импорта и т.п.
- добавить “маркер версии” в ответе на Excel, чтобы удостовериться, что код обновился
